# Vulkan ICD vs llama.cpp/ggml-vulkan advertised limits gap

Snapshot date: 2026-05-19.

Scope: documentation-only audit of the current Skydnir glibc Vulkan ICD surface
reported through `vkGetPhysicalDevice*` and the Vulkan properties/features that
current llama.cpp `ggml-vulkan` may consult while selecting queues, allocation
sizes, descriptor layouts, subgroup kernels, and Q6_K matmul/dequant pipelines.
The current ICD source of truth is `docker-proot-setup/src/gpu/pdocker_vulkan_icd.c`;
llama.cpp is not modified by this plan.

## Q6 / llama GPU bridge summary

The current advertised surface is intentionally compute-buffer-only and mostly
conservative.  It is sufficient for llama.cpp to enumerate the bridge, create a
compute device, allocate buffers, and attempt quantized Vulkan kernels.  The
highest-risk gaps for the Q6_K lane are not missing table entries but
optimistic capability claims that must match the Android executor's real device:
16-bit storage, 8-bit storage / shader int8, subgroup arithmetic, memory/buffer
caps, and descriptor count pressure.

Short next actions:

1. Keep `ngl=0` as the correctness boundary while validating advertised limits.
2. For each fresh `ngl=1` Q6 run, record ICD configured clamps, Android physical
   limits/features, enabled features, SPIR-V required-feature masks, descriptor
   counts, and first failed stage.
3. Do not enable `shaderFloat16`, subgroup arithmetic, integer-dot-product,
   buffer-device-address, or cooperative-matrix claims for product runs until a
   device artifact proves the executor can create and dispatch the same path.
4. If Q6 fails after setup masks match, treat it as Q6 shader/descriptor/writeback
   semantics; if setup masks differ, fix the advertised limit/feature contract
   before doing Q6 oracle analysis.
